# AsyncDownload 接手文档清单

## 1. 项目目标

- 项目名称：`AsyncDownload`
- 目标：实现一个跨平台的高性能 HTTP 异步下载库与 CLI
- 核心技术栈：`C++20`、`CMake`、`vcpkg`、`libcurl`、`concurrentqueue`、`nlohmann/json`、`GoogleTest`
- 当前构建目标：
- 静态库：`asyncdownload`
- CLI：`asyncdownload_cli`
- 测试：`asyncdownload_tests`

## 2. 来自历史线程的硬约束

- 网络层必须基于 `libcurl multi interface`
- `WRITEFUNCTION` 只能产出 `DataPacket`，不能直接修改 `RangeContext` 持久化状态
- 整体线程模型是 `1 + 1 + N`
- `Orchestrator` 负责调度、背压、事件循环
- `Persistence` 线程独占乱序重排、4KB 对齐写入、位图更新、VDL 推进
- 工作线程池只负责 `Flush`、`CRC32`、元数据 IO 等后勤任务
- 背压是主动、非阻塞式
- 高水位：`256MB`
- 恢复水位：`128MB`
- 对齐规则：`IO_ALIGNMENT = 4KB`
- 位图/调度颗粒度：`BLOCK_SIZE = 64KB`
- 单 range 空洞超过 `32MB` 时需要触发 gap pause
- 恢复链路依赖 `config.json + VDL + CRC32 样本`
- 退出顺序是强约束：先停网络，再清空乱序碎片，再最终 flush，再更新 VDL，再关闭文件句柄
- 内存会计必须统计 `payload + sizeof(DataPacket) + map node overhead`

## 7. 当前实现层面的已知判断

- `DownloadClient` 已经作为稳定外部入口存在
- `DownloadEngine` 已经承担 probe、恢复判定、调度、持久化接线和最终收尾职责
- 测试规范文档明确了重写验收标准，不只是“能下载完成”，还包括恢复语义、位图语义、背压语义、调度切分语义
- 当前代码库已经不是纯骨架阶段，而是“主链已经存在、但仍需继续收敛验证和性能确认”的阶段

## 8. 当前仍需重点核实的风险点

- `download_engine` 是否已经完整覆盖规格中的主动背压策略，尤其是 Top 20% 快速连接 pause/resume 逻辑
- `Persistence` 是否严格满足 4KB 对齐、尾块处理、VDL 推进与最终 flush 顺序
- `CRC32` 样本恢复路径是否已经完整接到 metadata resume 流程
- `gap pause`、`memory pause`、range 复用、work stealing 是否与规格完全一致
- CLI 配置读取、默认参数和 `configs/download_options.template.json` 是否完全一致
- 端到端恢复测试是否覆盖中断、重启、部分块已完成、metadata 不匹配等关键场景
- 构建环境对 `vcpkg` 的依赖在不同机器上是否稳定，参考：
- `docs/vcpkg-dependency-issues.md`

## 9. 建议的接手优先级

1. 先验证当前代码是否能稳定配置、编译、运行测试
2. 核对 `download_engine` 与规格书中背压、调度、恢复、退出顺序的偏差
3. 核对 `persistence_thread` 的对齐写入、尾块落盘、VDL 更新和元数据快照行为
4. 用现有测试和本地 HTTP 服务器先确认功能正确性
5. 基于下载参数做系统化性能基线测试，形成对吞吐、内存、活跃连接数、恢复开销的基线认知
6. 再考虑把 `Orchestrator` 从 `download_engine` 中进一步显式模块化，前提是当前功能已稳定
